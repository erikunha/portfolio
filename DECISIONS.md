- **2026-05-18** · **Tailwind v4 removed.** Vestigial since scaffold: zero `@apply`, zero utility classes used anywhere in components — `@import "tailwindcss"` was the only reference. Was contributing ~2.25KB gzip of Preflight reset into the render-blocking CSS bundle and forced `style-src 'unsafe-inline'` via runtime style injection. Reversal cost is low; cost of keeping was a CSP downgrade and a dep that did nothing for this codebase._
- **2026-05-18** · CSS architecture lock-in: **global CSS in 10 hand-rolled files**, BEM-ish naming, tokens centralized in `_tokens.css`, one entry point in `globals.css`. Explicitly rejected CSS Modules: per-component scoping would scatter the same volume across 18+ section files and break the architectural property (single grep-able surface) that made the audit above possible in <60 seconds. _Reversible at significant cost; revisit only if multi-app reuse or shared design-system extraction becomes in-scope (currently rejected in CLAUDE.md)._
- **2026-05-18** · `/api/ask` kill switch chosen as env-var (`ASK_ENABLED`) rather than Vercel Edge Config or Upstash Redis flag. - **2026-05-18** · Kill-switch semantics are "off-by-keyword" (case-insensitive trimmed set: `false | 0 | off | no | disabled`) rather than "off-by-strict-literal". Asymmetry rejects the "typos default to enabled" alternative as architecturally wrong for a kill switch — during a cost emergency a typo MUST still disable. False-positive disablement recovers in <90s; false-negative non-disablement is the exact failure mode the switch prevents. _Reversible by editing the OFF_KEYWORDS Set in `app/api/ask/route.ts`._

- **2026-05-18** · Mobile critical CSS inlined in `app/layout.tsx` even though `./globals.css` remains render-blocking. Mechanic: the inline `<style>` is in the document head before the external link, so the browser populates the style cache during HTML parse rather than waiting on the link. - **2026-05-18** · `/api/ask` Q+A logging shape locked: question truncated to 500 chars + answer to 1000 chars + meta (`{requestId, ts, ipHash, inputTokens, outputTokens, durationMs, status}`), 90-day TTL, IP hashed via SHA-256 + DEPLOY_SALT. Architect-review identified GDPR Art. 17 / LGPD Art. 18 right-of-erasure requirement; addressed by new `/api/log/forget` endpoint (POST `{requestId}` → idempotent DELETE across the last 90 day-partitions) + `X-Request-Id` response header + privacy notice on the /api/ask UI. - **2026-05-19** · `err:*` records (from `/api/log`) redesigned to omit `ipHash`. Alternative considered: change the key to `err:{date}:{ipHash}:{uuid}` so `/api/log/forget` could SCAN+DEL by ipHash on user request. Rejected: (a) the triage value of ipHash in `err:*` records is low — the rate-limiter already enforces per-IP quotas, and correlation between error bursts and a single IP is not a current operational need; (b) storing it requires extending the erasure flow, adding SCAN cost and complexity; (c) a hashed IP without the re-identification key is borderline personal data (GDPR Art. 4(1)) but erring on the side of not storing it is safer and simpler. Net result: `err:*` records are personal-data-free, `/api/log/forget` scope (ask:log:* only) now matches the privacy notice exactly. _Reversible: re-add `ipHash` to the record shape and update `/api/log/forget` to accept `{ ipHash }` with SCAN+DEL pattern._

- **2026-05-19** · `lighthouserc.json` updated: dropped 4 audit assertions invalidated by Lighthouse 12.x audit-name churn (`tap-targets` renamed/removed; `forced-reflow-insight` + `network-dependency-tree-insight` + `render-blocking-insight` are new "insight" audits whose default minScore=0.9 expectation is wrong because insight audits report informational data, not pass/fail signal). Do not leave `warn` indefinitely.**_
- **2026-05-19** · `DEPLOY_SALT` auto-handling — `lib/ip-hash.ts` now lazy-resolves the salt: `env → Upstash KV (auto-generated + SETNX-persisted) → 'portfolio' literal (non-prod only)`. Removes the manual Vercel UI step previously required (which broke automatic deploys: every fresh deploy hard-crashed at module-eval if the env var was unset). The auto-generated value is a random 32-byte base64 string; subsequent reads converge on the same value because SETNX guarantees only the first concurrent cold-start wins. Salt is stable across deploys until the Upstash record is wiped (preserving rate-limit accounting + Q+A correlations). Threat model intact — value is server-side-only, never in env/logs/exposed surfaces; attackers can't enumerate without compromising Upstash. Operator override still supported by setting `DEPLOY_SALT` explicitly. _Reversible: revert this commit + restore the fail-fast IIFE; manual Vercel UI step returns as a requirement._
- **2026-05-19** · CLS root cause + fix: `app/css/_crt.css` `crt-scan-beam` keyframes animated `top` (a layout property), firing a LayoutShift PerformanceEntry on every paint frame. On ubuntu CI runners (~30x slower than local M-series Mac, by reproduced devtools CPU throttle measurements), the 7-second animation cycle accumulated CLS 0.075-0.083 — consistently above the 0.05 budget across 3 runs. Local simulate-method LH showed 4e-6 because simulate uses a timing model, not real Chrome CPU throttle; `top` animations emit LayoutShift events at actual frame rate regardless of simulate settings. Switched to `transform: translateY()` (GPU-composited, skips layout entirely) with `will-change: transform` for immediate layer promotion. Static `top: -120px` preserves starting position; `translateY(calc(110vh + 120px))` matches original travel distance. Re-tightened `lighthouserc.json` `cumulative-layout-shift` from `warn` back to `error` — closes the BLOCKING CRITERION from the prior gate-patch ADR. The earlier PR #16 (`display: 'optional'` for mono font) was wrong about the root cause; the font change stays as a defense-in-depth but did not contribute to the CLS fix. _Reversible: revert this commit; gate falls back to warn._
