## 2026-05-19 — dom-size + unused-javascript investigation (PR branch: perf/dom-size-and-unused-js)

- **2026-05-19** · **dom-size root cause and partial fix.** Measured baseline: 1717 DOM elements (score 0.5 against 0.9 target, flagged in DECISIONS 2026-05-19 "gate config patch"). Root cause: five sections (ProjectsSection, GitLogSection, GuitarSection, VisaSection, Hero) rendered both a desktop and mobile DOM tree simultaneously, relying on CSS `display: none` to hide the inactive variant. On a desktop Lighthouse run, the mobile DOM trees added ~440 nodes of hidden elements. Fix: converted ProjectsSection, GitLogSection, GuitarSection, and VisaSection to async RSCs that call `detectMobileFromUA(headers())` — the same pattern Module.tsx uses for the section shell — and render only the matching branch. Measured post-fix: 1277 elements. Hero's dual render was not changed: it contains HeroBootAnimation and HeroSystemFailure client islands that depend on the desktop section being present for interaction-driven show/hide; the complexity of restructuring those islands is disproportionate to the remaining DOM savings (~30-50 nodes). _Score after fix: 0.5 (same Lighthouse bucket but 440 nodes fewer; Lighthouse's metricSavings scoring for dom-size is non-linear and doesn't move the displayed score at 1277 vs 1717 — both report 0.5). Gate re-tightened: `dom-size` changed from bare `"warn"` to `["warn", { "maxNumericValue": 1500 }]` — prevents regression into the "poor" zone (Lighthouse threshold) without requiring content removal._
- **2026-05-19** · **dom-size score 0.9 not achievable for this composition.** The portfolio has 18 content-dense sections totalling ~1277 nodes post-fix. Lighthouse's "good" threshold is <800 nodes. Reaching 0.9 would require removing approximately half the content sections — defeating the purpose of the portfolio as a hiring artifact. The correct framing: the dom-size metric is calibrated for typical SPAs with 1-2 views; a 18-section portfolio is an outlier the metric doesn't model well. The `maxNumericValue: 1500` gate catches actual regressions (someone accidentally doubling DOM) while acknowledging the structural reality. _Re-tighten to error only if architectural changes (e.g., section-by-section lazy loading via route splitting) bring node count below 800. Do not re-tighten to score >= 0.9 without content reduction._
- **2026-05-19** · **unused-javascript root cause and gate calibration.** Measured: 2 items, 43KB total wasted bytes. Both items are Next.js App Router `rootMainFiles` — framework bootstrap chunks that are unconditionally loaded before any user code: (1) `08i` chunk: ~70KB compressed, contains getDeploymentId, getAssetPrefix, InvariantError, and other Next.js initialization code — 31% unused on initial paint; (2) `0fjplx8p` chunk: ~37KB compressed, contains the App Router RSC client handler, Server Actions dispatcher, and client-side navigation runtime — 57% unused on initial paint. These chunks are required for RSC hydration and the full App Router runtime even though most of their code paths don't execute on the initial page load sequence (they're needed for navigation, form interactions, revalidation). There is no actionable fix: they cannot be dynamically imported (they run before any module load), and removing them requires abandoning App Router. The wasted-bytes figure is structurally stable — it reflects Next.js's framework bootstrap split, not application-level sloppiness. Gate set to `["warn", { "maxNumericValue": 60 }]`: the audit's `numericValue` is the overallSavingsMs estimate (measured at 40ms currently); budget of 60ms gives ~50% headroom before the gate fires, catching if framework bootstrap cost significantly regresses. _Do not re-tighten to `error` unless Next.js provides a mechanism to tree-shake unused App Router runtime paths.
