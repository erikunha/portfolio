## 2026-07-19 - The push-guard predicate is a tokenizer walk, and the property is execution, not spelling

- **2026-07-19** · **`.claude/hooks/api-security-push-guard.sh` no longer matches tokens; it consumes `bash-guard-detect.py --emit-commands`, the same `bashlex` walk `bash-guard.sh` already used to judge.** This is the follow-up the 2026-07-18 entry below names as not done — that entry's "Follow-up, not done here" is superseded here, and its reversibility note ("do not revert without first replacing the regex with the vendored `bashlex` tokenizer") is now satisfied. The raw-payload fallback and its deliberate over-strip survive unchanged, because the walk still reports undecidable (exit 3) for constructs it cannot resolve, and the fallback is where those land. **The governing design rule, which cost the most to learn: a construct the walk cannot resolve is undecidable, never clean.** Two mid-branch fail-opens were regressions where the precise detector was strictly *narrower* than the regex it replaced — precision is not safety.
- **2026-07-19** · **The predicate stopped enumerating dangerous spellings and now states one property: while an unaudited API edit pends, a git invocation must not gain configuration or execution it would not otherwise have.** Enumeration lost thirteen consecutive rounds — each named key (`alias.*`, `core.hooksPath`, `include.path`, `core.sshCommand`) was followed by another. The class that justifies an over-block is specifically the **no-artifact** one: a single Bash call that writes no file, so nothing else in the transcript records it. That covers config injection (`-c`, `GIT_*` env, `git config` writes including `--unset`, which removes `core.hooksPath` and disables husky exactly as redirection does) and **program injection after the subcommand** (`--upload-pack`, `--receive-pack`, `--exec`, `--extcmd`, `--template`, `--separate-git-dir`) — verified against git 2.53.0 to exec a named program against a local remote. An arg loop that stops at the first non-flag word cannot see any of the second group, which is why the scan is now whole-record. Artifact-requiring routes (`bash ./script.sh`, direct `.git/config` writes) are deliberately left to the artifact being the visible event; `-C` was relaxed back to skip-the-path for the same reason, since redirecting at a repo is useless without a config someone already wrote, and `git -C <dir> status` is ordinary worktree work. _Reversible: `git revert` restores token matching and reopens every under-match and over-match pinned in `.claude/hooks/__tests__/run.sh`; the suite fails loudly rather than silently._

## 2026-07-18 - push-guard fallback: the metadata strip errs toward over-blocking, deliberately

- **2026-07-18** · **`.claude/hooks/api-security-push-guard.sh`'s raw-payload fallback strips `transcript_path`/`cwd`/`description` with `"[^"]*"`, which stops at the first bare quote and therefore under-strips when a value contains an escaped quote — this is chosen, not overlooked.** The fallback only runs when the JSON extraction already failed, so every bound in it is an approximation over text that did not parse; the only question is which way it errs. `"[^"]*"` can strip **less** than intended, leaving residue in the haystack, which **over-blocks**. The escape-aware alternative `"([^"\\]|\\.)*"` can strip **more**: an odd trailing backslash makes the closing quote read as escaped, the match runs past the field, and a real `git push` is deleted — verified fail-open, `{"cwd":"x\", command git push, "y":"end"}` exits 0 with a marker pending. Over-blocking is recoverable (this is a PreToolUse hook on the agent's Bash tool; the owner pushes from a terminal it does not intercept); a silent bypass is not — once an unaudited API-surface push ships, no later action makes the audit have happened first. **This was implemented the wrong way first and reverted**: `ec635ae` shipped the escape-aware bound to close a reviewer-reported hard-lock, `16e409d` reverted it once the fail-open was demonstrated. The hard-lock (an escaped quote in prose leaves a token tail and blocks the marker-clearing escape) is now an accepted, tested behaviour rather than a latent bug. Both directions are pinned in `.claude/hooks/__tests__/run.sh`; re-introducing the escape-aware bound fails both. _Reversible: `git revert` restores the escape-aware bound and reopens the verified fail-open — do not, without first replacing the regex with the vendored `bashlex` tokenizer (see the follow-up below)._
- **Follow-up, not done here:** seven review rounds on this one predicate each widened a boundary or normalization and were broken by the next adjacency. The structural fix is to tokenize with the vendored `bashlex` that `.claude/hooks/bash-guard-detect.py` already uses (basename matching, `-c` re-parse, substitution/compound traversal), which closes both the under-match (`/usr/bin/git push`, `bash -c "git push"`, `g\it push` — all verified to exit 0 today, all backstopped by `.husky/pre-push` except under `--no-verify`) and the over-match (a commit message mentioning `git push` false-blocking).

## 2026-07-14 - The per-route JS budget was unreachable; corrected to a measured 175KB total, gated

- **2026-07-14** · **`CLAUDE.md` documented "JS gzipped per route < 120KB" as a hard, non-negotiable budget. That number was never achievable and had never been measured.** Nothing measured it because `scripts/check-bundle-size.mjs` sums *every* chunk in `.next/static/chunks` **globally** against one cap — it is structurally blind to a per-route figure, so a route reusing chunks that already exist for `/` adds ~0 new bytes and that gate reads green regardless of what ships. The first per-route measurement (`scripts/check-route-js.mjs`, added here) found `/` at 173.0KB and `/_not-found` — **a route with zero application code** — at **148.9KB**. A bare hello-world Next 16.2.10 + React 19.2.7 app on this toolchain, built for comparison, measures **142.2KB gzipped shared before a single line of app code exists**. Deleting 100% of this site's client JavaScript would land at ~142KB, still 22KB over the documented budget. **A 120KB per-route budget fires on an empty Next app**, which is exactly CLAUDE.md's stated exception for a legitimate gate-config change ("an assertion that fires on the ideal state"); every other reading of the standing rule would require deleting Next to satisfy it. Hypothesis tested and **refuted**: there is no dependency leak — `zod`, the `ai` SDK, `@upstash`, `resend`, and `@anthropic-ai/sdk` are absent from every client chunk (the `content/_validate-client-content.ts` pattern is holding), and `cacheComponents: true` costs only 0.7KB. **Replaced with ONE gate that measures a real property:** per-route **total ≤ 175KB** — the 142.2KB measured floor plus 32.8KB of app headroom. `/` is 173.0KB today, so the margin is ~2KB; note that a routine `react`/`react-dom` bump (caret-pinned, and the stack policy runs `pnpm up --latest`) can trip it with ZERO app-code change, since the framework runtime chunk alone is ~70KB — a dependency-bump PR should budget for re-measuring the floor or trimming app JS. **App-owned is REPORTED, never asserted, and that is deliberate:** it is `total − FRAMEWORK_FLOOR_KB`, an affine transform of total by a constant, so an app-owned threshold is a second threshold on the SAME axis. The first version of this ADR claimed a gated 43KB app-owned budget; it could only have fired above 142.2 + 43 = 185.2, which the 175 total already caught, so it was unreachable — a budget that printed a number no input could ever trip while the real ceiling was 32.8. Caught in review. **The absence of the second threshold is NOT held by a test, and cannot be:** re-adding `|| appOwned > 43` to the gate leaves exit code and stderr byte-identical for every possible input (mutation-tested in review — the mutant survives), precisely because the threshold is unreachable dead code. What `__tests__/check-route-js.test.mjs` does pin is that a route in the disagreement band (175 < total < 185.2) fails at all, which would catch someone reintroducing the two-gate design by relaxing the total to 185. That is worth having; it is not the same claim, and this entry previously conflated them. The app-owned figure is the one engineering moves; the framework floor is not ours to shrink. Enforced by `pnpm route-js-check`, wired into `ci.yml` (after the build, which emits the manifest it reads), the `ci` script, and `ready-for-pr`. **App-owned is deliberately NOT "the chunks a route does not share with its siblings."** That definition — the first one written here — fails open twice: with a single route the intersection is the whole bundle, so app-owned computes to 0 and the gate prints OK over an unmeasured budget (proven in review); and a client island moved into `app/layout.tsx` lands on every route, joins the "shared" set, and silently leaves the budget it exists to constrain. Both are pinned by `__tests__/check-route-js.test.mjs`. The floor constant is itself gated: if the lightest route drops below it, the gate fails and says to re-measure rather than let every app-owned figure inflate silently. **Also fixed here:** `scripts/strip-next-polyfills.mjs` stripped `polyfill-module.js` but not `polyfill-nomodule.js`, a 112,594-byte file the build shipped as a real chunk (**38.6KB gzipped**). No supported browser ever downloaded it — it is served with `noModule`, and every browser in this repo's browserslist (`chrome >= 93`, `edge >= 93`, `firefox >= 92`, `safari >= 15.4`) has had ES modules since ~2017 — but `bundle-check` summed it blindly, inflating that gate's reading to **214.5/220KB**. Stripping it drops the real figure to **175.9/220KB**: the gate was never nearly-blown, it was reading ~38.6KB of bytes nobody fetches. **Measured in isolation on 2026-07-16, not inferred.** Restoring only `polyfill-nomodule.js` into `node_modules` (pristine from the npm tarball for next@16.2.10, MD5 `846118c33b2c0e922d7b3a7676f81f6f`, 112,594B raw) and running `pnpm build && node scripts/check-bundle-size.mjs --max-client-kb=220` gives **214.5KB**; re-stripping (`node scripts/strip-next-polyfills.mjs`) and rebuilding gives **175.9KB**. Delta: **38.6KB**. Reproduced independently twice, including a from-scratch `npm install next@16.2.10`. For scale, the pristine file alone gzips to 39,627B = 38.698 KiB under `zlib.gzipSync(buf)` with no options — the exact call `check-bundle-size.mjs` makes. **The file figure and the build delta differ by ~0.1KB purely because both reported figures are already rounded**: 214.5 and 175.9 each carry ±0.05, so their difference carries ±0.1 and spans 38.50–38.70, which contains 38.698. No mechanism is needed to explain the gap, and two were proposed and discarded — "bundler wrapping makes the chunk differ" (wrong direction: wrapping adds bytes) and "other chunks' rounding drifts between builds" (unverified, and unnecessary once the arithmetic is done). **Cite the build delta, not the file size** — the delta is what the gate measures.

**An earlier revision of this entry said 219.2 → 175.9, a 43.3KB delta, attributed to a "39.5KB" polyfill. All three figures were wrong and none reproduce.** 219.2 does not reproduce under any measurement tried here; 39.5 has no recoverable provenance, and guessing at its origin would repeat the error that produced it.

The lesson worth keeping: **gzip is not one number, and the `gzip` CLI's number is not even one number.** A revision of this entry cited 39,478B from `gzip -c`. A reviewer could not reproduce it and got 39,494B. Both are correct: **the gzip CLI embeds the source filename in its header**, so the same 112,594 bytes compress to 39,478B as `u.js` (a 4-character scratch name) and 39,494B as `polyfill-nomodule.js` — exactly 16 more, the filename-length difference. `gzip -nc` strips the header name and both collapse to 39,473B; Node's zlib never writes a name and reports 39,627B. That 39,478 was reproducible only on the machine that happened to name its temp file `u.js`: **it encoded an accident of the measurement, not a property of the artifact.** Measure with the tool the gate runs (`zlib.gzipSync`), or the number is about your filesystem rather than your bundle. **Rejected:** removing `<Analytics/>` + `<SpeedInsights/>` from the shared graph (2.3KB) — it does not bring `/` under any budget it fails, and it would cost the real-user Core Web Vitals data this site publishes as a feature. **Rejected:** switching to `--webpack` (~19KB) — the build currently fails under webpack (MDX design-system pages export `metadata` from `'use client'`) and it abandons the supported Next 16 bundler. _Reversible: restore the numbers in `scripts/check-route-js.mjs`. Do not re-tighten below the floor without re-running the hello-world measurement — the number is evidence, not preference._

The pointer path is additionally session-pinned (claude-review finding, PR #183): the output file must end with `/<sessionId>/tasks/<taskId>.output` where sessionId derives from the transcript filename (harness-provided input, charset-validated, fail-closed when absent) — this rejects planted files at arbitrary roots matching the generic suffix AND cross-session replay of a genuine old PASS output file. The sync tool_result path is unchanged. The pointer is also ordering-bound (second claude-review finding, PR #183): `task-id` and `tool-use-id` arrive in the same attacker-suppliable string with nothing binding them to one dispatch, so a forged notification pairing the CURRENT dispatch id with an EARLIER same-session genuine PASS file was a zero-privilege intra-session replay. The gate now records the governing (latest) architect dispatch record's timestamp and requires the output file's mtime to be strictly after it (`TaskOutputReader` returns `{content, mtimeMs}`; missing dispatch timestamp or mtime fails closed) — a stale file always predates the dispatch it is being replayed against. The pointer is finally provenance-bound (battery security finding, same PR): mtime proves FRESHNESS, not IDENTITY — a pasted pointer could still name a fresher SIBLING subagent's output file that merely quotes the sentinel, and that is not hypothetical (measured: two unrelated battery outputs in this session each contained `GATE_RESULT: PASS` verbatim, and the pre-fix gate ACCEPTED the substitution against real harness files; post-fix it rejects it while the genuine file still passes). The gate now requires the named output file to carry the architect dispatch's OWN prompt (first 200 chars of `input.prompt`, read from the dispatching `tool_use` record — structured blocks in an assistant message, which typed chat text cannot produce), fail-closed when the dispatch has no bindable prompt. Known cost: this couples the gate to the harness writing the dispatch prompt into the task output file; if that stops, the gate fails CLOSED (blocks writing-plans) rather than open. The verdict scan is scoped to the assistant's OWN text blocks (battery code-review finding from #183's own provenance review — the finding arrived after #183 had already MERGED, and is fixed forward in PR #186): the needle was matched against the whole file, but the harness echoes the dispatch prompt into the file's first record and architect prompts literally instruct "return GATE_RESULT: PASS or GATE_RESULT: FAIL" — so every accepted file contained the sentinel in its prompt echo and the gate accepted a real `GATE_RESULT: FAIL` as PASS (reproduced against real harness data; no attacker needed). The gate now requires `GATE_RESULT: PASS` in an `assistant`-role text block (thinking blocks carry `.thinking`, not `.text`, so deliberation is excluded), where the verdict actually lives; provenance (prompt anchor) and freshness (mtime) still gate the same file. That first cut still matched the sentinel as a SUBSTRING of the assistant block, leaving a narrower residual of the same class (battery security finding): the architect's verdict template is `GATE_RESULT: PASS | BLOCK` and its negative verdict is `GATE_RESULT: BLOCK`, so a genuine BLOCK whose prose quotes the PASS token or echoes the template line still read as PASS (no attacker; the architect's own phrasing triggers it). The match is now the LAST standalone verdict line: a trimmed line equal to the needle, taking the last line that begins with the verdict key (`GATE_RESULT: `) as the decision so a later `GATE_RESULT: BLOCK` overrides an earlier stray PASS line (last-verdict-wins). The same line-exact check replaced the sync `tool_result` path's whole-blob `JSON.stringify(item).includes(needle)`, which shared the substring fail-open. Reproduced against realistic architect-shaped data: a PASS with the verdict on its own line corroborates; a BLOCK that both quotes the PASS token in prose and echoes the `GATE_RESULT: PASS | BLOCK` template is rejected. **Inherent limitation (honor-system boundary):** any actor with file-write capability (Bash/Write) can still author a prompt-carrying, PASS-bearing file at the real session task root; that capability is inside the gate's own TCB (equivalent to editing the hook itself), same class as review-stamp's "cannot know about a finding you never recorded". Text-only injection — the class this ADR exists to close — cannot manufacture a corroborating artifact (it cannot create a file, forge a `tool_use` dispatch record, or cause a sibling agent to emit both the architect's exact prompt prefix and a PASS verdict in its own assistant output). _Reversible: revert `scripts/lib/transcript.mjs` to a record-shape anchor — but expect it to break on the next CLI transcript-shape change; the file model is shape-independent._

## 2026-06-05 — Font-loading: mono `font-display: swap`, and `@theme` tokens single-sourced to next/font vars

- **2026-06-05** · **Two coupled font-loading decisions (PR #98).** (1) **`font-display: optional` → `swap` on the self-hosted JetBrains Mono.** `optional` gives a ~100ms block period and NO swap period, so on a cold cache the woff2 missed that window and the browser committed to the metric-matched fallback for the entire first pageview — the custom font only appeared after a reload had cached it (the reported "wrong font on first load, correct after refresh" bug). `swap` paints the fallback immediately then swaps in the real font on load, so the first visit always lands on JetBrains Mono. The metric-matched next/font fallback keeps the swap reflow inside the CLS budget (measured ≤0.0439 vs 0.05; cold-cache + Slow-3G reproduction confirms the real face loads and swaps). (2) **`@theme` font tokens now reference next/font's `--font-*-src` vars instead of hardcoded family names.** `theme.css` previously set `--font-mono: "JetBrains Mono"` / `--font-display: "Geist Black"`, but neither literal is a loaded `@font-face` (next/font names its faces after the variable — `mono`/`display` — and the display font is actually Inter, not Geist). The custom fonts rendered ONLY because next/font's same-named override won the CSS cascade against those dead literals; a future Next/Tailwind injection-order change could silently flip that to system fonts on every load. Trade-off: HTML response grows by ~12KB gzipped (~70KB uncompressed); for a portfolio with low repeat-visit rate, the loss of independent CSS caching is acceptable. The residual blocking link is now a 1KB font-CSS chunk from `next/font/local` (88% smaller than the 10KB globals.css chunk). Measured localhost contribution: LCP 3066 ms → 2924 ms (-142 ms, spread 12 ms). `lighthouserc.mobile.json` LCP threshold ratcheted from 3400 ms to 3066 ms. The CSS-inline minifier is a regex pass (strip comments, collapse whitespace, remove punctuation spacing) — sufficient to clear Lighthouse's `unminified-css` audit without adding a dep. Drift-protected by `__tests__/inline-css.test.ts` (per-file selector fingerprints + order + LCP-element selector). _Reversible: restore `import './globals.css'` + the prior `CRITICAL_CSS` partial in `app/layout.tsx`; the chunked link returns and the inline `<style>` becomes redundant overhead._
