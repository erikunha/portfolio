**Date:** 2025-05
**Scope:** Full codebase — every component, API route, lib, content layer, config

## Summary

The RSC composition model is structurally sound: all 18 section components are RSC by default, with client islands only where required. The security headers, rate limiting, and error boundary patterns are production-grade. The content/Zod layer is correctly wired for most sections. The font/motion/hydration fixes from the previous session are all clean.

**Four issues require action before launch.** The rest are improvements with trade-off notes.

## CRITICAL

### A-01 — `LivePerfSection` blocks page render with no `<Suspense>` boundary

**File:** `app/page.tsx`, `components/sections/LivePerfSection.tsx`, `lib/lighthouse-scores.ts`

`LivePerfSection` is an async RSC that calls `getScores()`. On a Redis cache miss (cold start after deploy, or 24-hour TTL expiry), `getScores()` calls the PageSpeed Insights API — which can take 8–30 seconds. There is no `<Suspense>` wrapper in `page.tsx`.

**Impact:** On production with a warm Redis key, this is invisible. On cold start, the entire page render holds until PSI responds. LCP spikes to 30+ seconds, breaking the < 1.8s budget. This is the highest-risk item in the codebase.

**Also:** `getScores()` calls `fetch(psiUrl, { next: { revalidate: TTL_S } })` with no abort signal. A 30-second PSI timeout holds the Vercel Edge invocation open.

**Fix:**

1. Add `AbortSignal.timeout(8_000)` to the PSI fetch in `lib/lighthouse-scores.ts`:
   ```ts
   const res = await fetch(psiUrl, {
     next: { revalidate: LIGHTHOUSE_TTL_S },
     signal: AbortSignal.timeout(8_000),
   });
   ```

2. Wrap `<LivePerfSection />` in `page.tsx` with `<Suspense>`:
   ```tsx
   import { Suspense } from 'react';

   <Suspense fallback={<PerfSkeleton />}>
     <LivePerfSection />
   </Suspense>
   ```
   A `PerfSkeleton` can be a simple static placeholder with the same `Module` chrome.

### A-02 — `?force=desktop` has no exit path — user is trapped

**File:** `components/PageChrome.client.tsx`, `components/responsive/MobileTitleBar.tsx`

`MobileTitleBar` renders only when `isMobile === true`. It contains the only "DESKTOP ↗" switch (`href="?force=desktop"`). Once `?force=desktop` is set, `isMobile` becomes `false`, the title bar disappears, and there is no UI control to return to mobile layout. The user must manually edit the URL.

**Fix:** Add a "↙ MOBILE" link to `DesktopTopbar` that removes the `force` param:
```tsx
{forceDesktop && (
  <a href="/" className="topbar__mobile-back">↙ MOBILE</a>
)}
```
`DesktopTopbar` needs to receive `forceDesktop` as a prop (currently it doesn't). Alternatively, read `window.location.search` and conditionally render.

## HIGH

### B-01 — Dual `MatrixRain` canvas — two rAF loops on every page

**Files:** `components/sections/Hero.tsx`, `components/PageChrome.client.tsx`

`Hero.tsx` mounts its own `<MatrixRain>` instance (inside the hero section). `PageChrome.client.tsx` mounts a second, full-screen `<MatrixRain>`. Both run their own `requestAnimationFrame` loops at `1000/22 ≈ 22 fps`. On desktop with a GPU this is imperceptible. On mobile (where `isMobile` is true) both loops fire simultaneously at 22fps across two canvases.

Two active rAF loops processing canvas 2D operations are the single largest INP and battery-drain risk on mobile — particularly harmful on sub-$400 Android devices where job recruiters are reading the portfolio.

**Fix:** The Hero canvas should not run when the full-screen background canvas is already running. Options, ranked:

1. **Preferred:** Pass a `watchRef` to the Hero's `MatrixRain` that points to the Hero section element. `MatrixRain` already accepts a `watchRef` prop that pauses when not intersecting. The full-screen canvas is always running, so the Hero canvas is redundant. Consider removing it and letting the full-screen canvas bleed through the Hero's transparent background.

2. **Alternative:** Hero keeps its canvas, but `PageChrome` pauses the full-screen canvas when Hero is intersecting (inverse watchRef logic).

### B-02 — `ManPageSection` ignores `content/man-page.ts` — full content inline in JSX

**File:** `components/sections/ManPageSection.tsx`

`content/man-page.ts` exists but `ManPageSection` does not import from it. All man page copy (NAME, SYNOPSIS, OPTIONS, EXAMPLES, ENVIRONMENT, BUGS, SEE ALSO — multiple scrolled screens) is hardcoded as JSX strings directly in the component.

This is the largest content discipline violation remaining. The file was created but never wired up. Any copy edit requires touching a component file instead of a content file, and the Zod guard is bypassed.

**Fix:** Import and render from `content/man-page.ts`. The schema should cover section labels and their raw text, which gets rendered with pre-computed span classes.

### B-03 — `GitLogSection.COMMITS` and `SysHealthSection.STATS` are inline — no Zod guard

**Files:** `components/sections/GitLogSection.tsx` (COMMITS array), `components/sections/SysHealthSection.tsx` (STATS array)

`GitLogSection` has an 8-entry `COMMITS` array inline (career history as git log — distinct from the `employers` blame view which was already wired to content files). `SysHealthSection` has 4 hardcoded stat cards: "8+ YRS", "40M+ tx/yr", "~100/100 a11y", "-33% JS / -98% CSS".

These are the highest-visibility quantitative claims on the page. They have no Zod schema, no build-time validation, and no single source of truth. If the Betsson numbers change, you edit a component instead of a content file.

**Fix:** Move both to content files. COMMITS → `content/git-log.ts` (or merge into `content/employers.ts` since it's the same career history, different view shape). STATS → `content/sys-health.ts`. Add Zod schemas for each in `content/schemas.ts`.

## MEDIUM

### C-01 — `getScores()` Redis singleton is inside a non-client-guarded module

**File:** `lib/lighthouse-scores.ts`

Minor: `_redis ??= Redis.fromEnv()` is a module-scope singleton. This is correct for serverless warm-invocation reuse. No issue. Note only: if `UPSTASH_REDIS_REST_URL`/`TOKEN` are absent (CI, preview deploys), `Redis.fromEnv()` will throw at first call. The `getRedis().get().catch(() => null)` guard in `getScores()` correctly catches this. The PSI path correctly short-circuits on missing `PSI_API_KEY`. **No action needed** — just documenting it's guarded.

### C-02 — `InteractiveShell` uses module-scope mutable `_lineId` instead of `useRef`

**File:** `components/client/InteractiveShell.tsx`

`let _lineId = 0` at module scope. In the browser with a single shell instance, this is harmless. If a future test mounts two shells in the same module scope (e.g., before/after tests without module isolation), they'd share the counter and line IDs would collide in React's reconciler.

**Fix:** `const lineIdRef = useRef(0); function nextId() { return ++lineIdRef.current; }` — moves the counter to be per-instance and React-idiomatic.

### C-03 — `Dock.tsx` is a client component without `'use client'` directive

**File:** `components/responsive/Dock.tsx`

`Dock` uses `document.getElementById`, `window.dispatchEvent`, and `React.MouseEvent`. It works because it's rendered inside `PageChrome.client.tsx` (an explicit `'use client'` boundary), but the directive is absent from `Dock.tsx` itself. This violates the project convention: client components should be visually identifiable as such. Future refactors could move `Dock` outside the boundary and silently break.

The file is also not named `Dock.client.tsx` per CLAUDE.md convention, though the naming convention appears to apply selectively (only files that ARE the boundary entry point use `.client.tsx`). The more important fix is the explicit directive.

**Fix:** Add `'use client';` as the first line of `Dock.tsx`.

### C-04 — `CRTOverlay` duplicates the inline IIFE motion init

**File:** `components/responsive/CRTOverlay.tsx`

`CRTOverlay`'s `useEffect` calls `applyMotion(readMotion())` on mount. The inline `<script>` in `layout.tsx`'s `<body>` already sets `body[data-motion]` before paint. The mount call in CRTOverlay is a synchronous no-op on every hydration.

The `mq.addEventListener('change', handler)` that follows is the valuable part — it keeps the motion preference synced to OS-level changes while the tab is open.

**Fix:** Remove `applyMotion(readMotion())` from the initial mount call in CRTOverlay. Keep the `mq.addEventListener` listener.

```tsx
useEffect(() => {
  const mq = window.matchMedia('(prefers-reduced-motion: reduce)');
  const handler = () => applyMotion(readMotion());
  mq.addEventListener('change', handler);
  return () => mq.removeEventListener('change', handler);
}, []);
```

### C-05 — `opengraph-image.tsx` uses system monospace, not JetBrains Mono

**File:** `app/opengraph-image.tsx`

The Edge runtime used by `ImageResponse` can't access `next/font/local` bindings. The OG image falls back to `ui-monospace, "Courier New", monospace`. On most systems this renders Courier New, which is visually inconsistent with the portfolio brand.

**Fix:** Load JetBrains Mono via the `fonts` option in `ImageResponse`:
```ts
const font = await fetch(
  new URL('../../public/fonts/jetbrains-mono-700.woff2', import.meta.url)
).then((res) => res.arrayBuffer());

return new ImageResponse(<...>, {
  ...size,
  fonts: [{ name: 'JetBrains Mono', data: font, weight: 700 }],
});
```

### C-06 — `sitemap.ts` uses `new Date()` — recrawls on every deploy

**File:** `app/sitemap.ts`

`lastModified: new Date()` generates the current timestamp at build time. Every deploy creates a new sitemap date, signaling to Googlebot that the page changed — even if nothing on it did. This wastes crawl budget and may inflate crawl frequency.

**Fix:** Use a fixed ISO date string that you update manually when content actually changes, or read a `CONTENT_UPDATED_AT` env var set in the Vercel deployment config:
```ts
lastModified: process.env.CONTENT_UPDATED_AT
  ? new Date(process.env.CONTENT_UPDATED_AT)
  : new Date('2025-05-01'),
```

### C-07 — CSP `'unsafe-inline'` on `script-src` — documented tradeoff

**File:** `next.config.ts`

The inline `<script>` tags in `layout.tsx` (scroll restoration reset, motion init IIFE) require `'unsafe-inline'` in `script-src`. This allows any inline script on the page to execute, which weakens XSS mitigation (though Next.js's output escaping and RSC serialization make script injection very difficult in practice).

The more secure alternative is per-request nonces: generate `crypto.randomUUID()` in a middleware, set it as a response header, pass it to the inline scripts and to the CSP `script-src 'nonce-...'`.

For a portfolio site with no user-generated content, this is an acceptable tradeoff. **Action:** Add an entry to `DECISIONS.md` documenting the CSP inline-script choice and the nonce alternative.

**File:** `components/responsive/StatusBar.tsx`

**File:** Missing

### D-03 — `ask` streaming truncation is silent on Anthropic mid-stream error

**File:** `app/api/ask/route.ts`

If Anthropic throws after the stream has started (e.g., content policy, mid-stream context overflow), `controller.close()` in the `finally` block closes the stream cleanly. The client (`InteractiveShell`) reads the truncated text as a valid complete response — there's no way to distinguish "stream ended normally" from "stream was cut short."

For the current use case (short Q&A, 512 max tokens) this is very low probability. Low priority.
